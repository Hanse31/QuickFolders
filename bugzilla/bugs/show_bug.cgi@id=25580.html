------- Comment #5 From Axel Grude 2013-09-20 00:05:52 ------- 
thanks for the additional detail. While testingwith SavedPassword editor my
initial suspicion is confirmed: the page has 2 login forms, which both use the
same field names (emailOrPcrNumber and pin). While QuickPasswords uses some
heuristic methods to prevent it from filling the wrong form, it is not always
possible to guess which of the forms in the DOM tree is the correct one. 

What I do is traverse all DOM nodes in the document and then checking the
element and all parent elements for visibility; if it or any parents are
hidden, I try t move on and find the next element of the same name. If I do not
find any, I return to the first element and fill that.

My guess is that in your case the "visibility check" has failed and
QuickPasswords has wrongly assumed the first matching textboxes (which you
cannot see) to be the correct form; it then proceeded to fill in your username
/password information (into the textboxes that arenot visible to you) and gave
you the message about successfully doing this. 

The problemn here is that software usually has no concept of what is rendered
on your screen (it cannot "see" your screen the way you do with your eyes)- it
has to make an educated guess by using said heuristics. I will attempt to
improve the algorithm based on the page and hopefully post an update shortly.
Unfortunately there is no standardized way of presenting login information, so
web designers can come up with various "creative" ways of showing login forms,
including completely hiding the input boxes from the DOM by using flash or
other hand-created input methods, so this will always make both the login
process and the "save suggestion" by Firefox process largely guesswork.

By the way you can verify this is the case with SPEby clicking Edit > Guess
From Current Page and then press "Next Forms" twice: you can see that SPE's
heuristics detect 3 possible candidates and the first and third have identical
username / password ids.

looking at the page with inspectorI foundthe first login form which is not
marked as invisible nested in one parent div "UHF_signInDialog" which has a
Z-Index of 6000:

<div id="UHF_signInDialog" class="UHF_pcrDialog UHF_accessibleHide"
tabindex="0">

I first tried increasing Z-Index as I suspected the designers used this to
"lower" the container below the one you are seeing but didn't get any visible
result. In actual fact here Z-Index is used in the opposite manner: to make the
login form float on top ofthe rest iof the page;  however the key is the class
UHF_accessibleHide, which has one rule that makes it go "off screen":

left: -9999em;

To use large pixel values to move elements offscreen is another common way of
"hiding" screen elements from the user; I will try to add detecting this to the
heuristics method of QuickPasswords. The author of SPE could do the same,
although his method might be to ignore the numerous possible screen / layout
properties and leave this to the better judgement of the users (therefore the
[Next form] button) - creating heuristic algorithms that are _robust_ for this
sort of thing is not always easy.

Try this test version, I have added some heuristic method that evaluates the
top and left computed properties and says the element (or parent) is invisible
when it is smaller than -200 (px or em). THis is an arbitrary value but I hope
it doesn't lead to false positives on any of your other sites. However it seems
to work with IHG.

To take a look at the functionality "under the bonnett", you can go to QP
options, enable debug mode and then right-click the debug checkbox to switch on
the following toggles:

extensions.quickpasswords.debug
extensions.quickpasswords.debug.formFill

you can see how the Add-on traverses the DOM tree and comes up with 

QuickPasswords {FORMFILL} 10:39:54.859  [0 ms]   
not filling - invisible: pin

QuickPasswords {FORMFILL} 10:39:54.859  [0 ms]   
not filling - invisible: emailOrPcrNumber

which means the elemnets were found once, but not filled as they were
determined as invisible.
