------- Comment #2 From Axel Grude 2012-08-20 05:47:28 ------- 
Created an attachment (id=7342) [details]
How to check your config settings

The conversion process should have copy the old configuration branches

extensions.smartTemplate.*

to new ones:

extensions.smartTemplate4.*

You can check this in tools > options > Advanced:
on the General tab there is a button [Config editor...]

(see attached image)

If you enter smartTemplate in the search part at the top of the dialog, you
should get a bunch of settings start with extensions.smartTemplate4.id

these are copies of the old settings starting with
extensions.smarttemplate.id

the reason this change had to be done is that the original name of the
extension was smarttemplate and this was changed ages ago and needed to be
brought to compliance. If your st4 settings are missing then obviously there
must have been a bug during the import.

You have 2 things that you can do - temporarily roll back to the older version
of st4 (0.8.5.6) by installing it from

https://addons.mozilla.org/de/thunderbird/addon/smarttemplate4/versions/

Or try to re-run the import by resetting

extensions.smartTemplate4.firstRun to true
and
extensions.smartTemplate4.version to ?

(simply double click these settings to reset them to defaults)

this will attempt a fresh import of the old settings when you restart Tb.
During this, the old templates are never deleted.

In order to FIX the problem, I would be very interested in what might have
caused this. If you like to try re-importing your old settings, please open

Tools > Error Console

and click on Reset (Delete all output of the error console), and then enable
all error logging

_before_ you click on the Ok. Hopefully an error will be shown which will make
it possible for us to fix this issue.

------- Comment #6 From Axel Grude 2012-08-20 23:28:51 ------- 
Ok, I understand now. So the conversion process itself did not simply copy your
templates, it must have done something wrong while reading them. This can
happen if a string is read with the wrong encoding, and might potentially
affect other languages as well. Unfortunately reading strings from an
international user base is a much less trivial thing than just simple 8bit
AscII.

It would be helpful if you could attach one of your original templates for
testing. You can get them in 2 ways:

1 - open the config editor
Tools > options > Advanced:
on the General tab there is a button [Config editor...] (see comment 1)
then filter for extensions.smarttemplate.*.*msg
you can then right click each template and select copy
paste them to a text file that you can attach to the bug

2 - directly from the prefs.js file
Open your profile folder
In XP this would be in
C:\Documents and Settings\userName\Application
Data\Thunderbird\Profiles\profileName.default\chrome

(userName will be your logon name, profileName will be random characters)
find all lines starting with
user_pref("extensions.smarttemplate.fwdmsg"
user_pref("extensions.smarttemplate.rspmsg"
user_pref("extensions.smarttemplate.newmsg"

Copy them together in a text file and attach them to the bug.

Even one of each would probably be sufficient. Thanks for your help!
------- Comment #7 From Axel Grude 2012-08-27 15:57:24 ------- 
Created an attachment (id=7362) [details]
new version with improved import and test function

this is a special test version I developed based on a Cyrillic template from a
Russian user. You can actually test the output of your old smarttemplate
settings with a hidden function that can be accessed via click on the
SmartTemplate logo on the about page.

After clicking on this logo, you can enter the name of one of the template
settings, e.g.

extensions.smarttemplate.id1.rspmsg
extensions.smarttemplate.id1.fwdmsg
extensions.smarttemplate.id1.newmsg

after this st4 will load the settings from the config database and display the
converted version of the template (which is what it would save to the regular
new preference branches)

extensions.smartTemplate4.id1.rspmsg
extensions.smartTemplate4.id1.fwdmsg
extensions.smartTemplate4.id1.newmsg

this should be sufficient to test how st4 would act during update from 0.8 to a
later version. If this is working as expected, then it is possible to reset the
config settings to force a new import of the old settings with this new
version.
