Description:   Opened: 2014-07-15 19:07 
Have two filters.

The first looks for "[SITE ERROR]" and 127.0.0.1 using Match All (i.e.,
development errors).

The second just looks for "[SITE ERROR]" (i.e., real errors).

Find Duplicate Attributes does not factor in the Match All filter option when
locating duplicates and flags the two instances of "[SITE ERROR]" as duplicate.

Not a high priority as most people do not use the Match All option as I am.

(P.S. Great product!)

> Find Duplicate Attributes does not factor in the Match All filter option when
> locating duplicates and flags the two instances of "[SITE ERROR]" as duplicate.

Yes this behavior is by design. I probably WONTFIX it. The idea comes from the
requirement of a user who had filters based on lists of people (FROM field) and
he wanted to avoid the same person appearing in 2 different filters. 

The algorithm  looks for all duplicates across all filters using a "cartesian
join", that means it puts everything into one pool to find all duplicates. It
even finds if you have the same condition twice within one filter.
------- Comment #2 From Chris Daley 2014-07-15 21:52:35 ------- 
Filter #1

 Match ALL of the following.
 - Subject contains "[SITE ERROR]"
 - Body Contains "127.0.0.1"
 ... Move Message to "No Big Deal" Folder
 ... Stop Filter Execution


Filter #2

 Match ANY of the following.
 - Subject contains "[SITE ERROR]"
 ...Move Message to "We Have A Problem" Folder


Incoming messages from my laptop development environment (localhost) trigger
filter #1. Messages from my web server (not localhost) trigger filter #2.

Filter #1 is my only filter that uses the "Match ALL" option -- all the rest
use "Match ANY". I would have no problem if you decide to not to "fix" this
issue. 

I wanted to make sure you were aware of the issue, and maybe add a note in the
documentation that if you're an advanced user, "Find duplicate attributes" may
flag duplicates that should be left as is.

Setting this to later: There might be some merit in an option to ignore
duplicate filter conditions when they are in a "Match All" filter, but this
would probably need to be optional behavior (so it needs an additional option
in the Advanced settings of quickFilters options). I changing the bug status
from WONTFIX to LATER for now; if it is not too difficult [which I can't tell
right now] I might have a crack at this over the next couple of weeks.
