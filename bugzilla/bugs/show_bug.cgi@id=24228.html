------- Comment #1 From Axel Grude 2011-07-23 11:31:13 ------- 
First off all, the error console out put you sent isn't conclusive of a bug,
does the slowdown get progressively worse? This would point to a recursion or
memory hole issue. Also, can you keep an eye on how much RAM Thunderbird takes
up, is it constantly increasing and never dropping down again?

Next question is whether you are using IMAP or POP3 (My guess is IMAP), and if
it is IMAP, how many folders have you subscribed to?

It would also be helpful if you could open DOMI (DOM inspector). use File /
Inspect Chrome > ... Thunderbird and  then Edit / Select Element by click and
then click on one of the QuickFolder tabs. What do you see under the "class"
heading, is this very long and has a lot of repeating items (such as pastel) in
it?

Also, how does the slowing down manifest itself? Is it a slow preview of the
emails, or slow changing of the focus when you go to the next email (so is
changing between emails in the same folder slow or is changing to a different
folder slow) ... thanks for your observations.

------- Comment #2 From Mark Rousell 2011-10-14 09:39:46 ------- 
Hello, I am getting the same problem on TB7.0.1 on Win7 Ult 64 bit with
Quickolders 2.9.1. First of all I'll describe the nature of the slow down that
I experience and then the specific conditions under which it occurs.

Nature of the slow down. It is specifically noticeable in three places:
(1) In opening TB's menus. They are slow to draw taking a noticeable mount of
time to draw on screen.
(2) In switching from message to message in the current folder. It takes a
noticeable amount of time for the next message to replace the previous message.
(3) Opening new windows (e.g. to write a new message or reply to a message) is
noticeably slow.

The conditions under which this slow down occurs. It does not occur all the
time that QuickFolders is loaded. It only occurs when the QuickFolders toolbar
has to wrap the folder tabs over more than one row. If there is only one row of
folder tabs then all operations are at normal speed. But if there is more than
one row then the above described slow downs occur.

As an example, if I start off with TB maximised with say 16 tabs (including the
Recent Folders tab) which take up one row then everything is fine, everything
works at the normal speed. If I now restore the TB window such that
QuickFolders has to wrap the tab over two rows, the slow down occurs. If I then
maximise the TB window again so that all the QuickFolders tabs can fit back on
a single row then the slow down ceases and everythng is back to normal speed.
