Description:   Opened: 2017-01-29 02:20 
The new Multiprocess Browsing feature titled "electrolysis" may break any way
that QuickPasswords currently changes / affects the content of any Firefox
browser tab. For the moment we can enforce disabling this feature by using a
flag in the build script for our Addon.

However future versions of Firefox may ignore this flag and enforce
MultiProcess browsing, thus breaking a lot of legacy Addons and only preferring
the (more chrome -similar) way that web extensions do their operations. It is
still not clear to me whether the current functionality (especially using the
logins list from a modeless window) can even bne implemented using Web
Extensions technology. But even if it is it will very likely necessitate a
massive rewrite.

As long as the XUL window for the logins will be maintained within Firefox
QuickPasswords will at least have a way of interacting with the logins in the
current way (yes it can be rewritten as long as access to XPCOM isn't also
deprecated); the barrier into the content window can probably be worked around
by implementing a messaging system which may be a few days/weeks work.

If you are a Addon-developer reading this bug, I would gladly pay you for
supporting me in that task in order to stay compatible for a longer while. For
the momentg, adding the flag should at least help in the currentFirefox version
(51). In the long run, a more permanent fix may be changing to Firefox Pale
Moon if Mozilla insists that XUL / XPCOM / privileged Addons should stop to
exist.

QuickPasswords 3.8 rev 22

Tested in Nighly 51.b07 this version carries out the login and also the "insert
password" from the context menu if this fails; field correction via
notification is also included.

to install, set:
xpinstall.signatures.required = false

to make e10s stays disabled, set:
browser.tabs.remote.autostart.2 = false
