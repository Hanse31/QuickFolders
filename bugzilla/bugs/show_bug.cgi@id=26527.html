Font setting with SmartTemplate4 on and disabled

The defalt composition that I set in "Thunderbird - Options - Composition -
General - HTML - Font" doesn't work when SmartTemplate4 1.5.1 is on. The
composition font is always "Variable Width" whatever I set in the Options. When
SmartTemplate is disabled, the composition font returns to what I set in the
Options.
------- Comment #1 From Axel Grude 2018-05-10 22:39:45 ------- 
I can try to fix this - but the way the font selection is implemented in
Composer is less than ideal. Basically Composer will insert a fresh <font> tag
wherever you start typing.

<font> is actually a deprecated tag, a superior way of setting the font for the
whole email / or paragraphs would be by adding in a <style> rule that applies
to <body> and <p> elements. This way much cleaner and more robust HTML markup
is created. 

Example for bad HTML:
    <p><font face="Arial">Dear John</font></p>
    <p><font face="Arial">how is the weather in L.A.?<br></font></p>
    <p><font face="Arial">cheers,<br>
 &nbsp; Axel<br></font></p>

Instead it is much better to just stipulate that the font is used for all <p>
elements.
    <p>Dear John</p>
    <p>how is the weather in L.A.?<br>
    </p>
    <p>cheers,<br>
      &nbsp; Axel</p>

So as a better solution I would highly recommend to add the following rule into
the Template:

<style>
p, body {
  font-family: Arial,Helvetica,sans-serif;
  font-size: 11pt;
}
</style>

Also make sure to disable the SmartTemplate option "Replace line breaks with
<br>" because it's very hard to write proper HTML templates with this enabled.
For easier editing & understanding the markup I highly recommend installing
Jorg Knobloch's Add-on "ThunderHTMLEdit" which will add a HTML tab to Composer.

If you care about fonts, I also recommend the setting Composition / General /
Use Paragraph format instead of Body Text by default. It is not a good idea to
have non HTML text at the body level as it tends to be hard to control how the
body level looks when your Email is quoted in a reply. Using <p> paragraphs and
adding style to all <p> elements as shown above is more reliable and easier in
the long run.
------- Comment #2 From Axel Grude 2018-05-10 23:21:00 ------- 
TECH NOTES

The font in composer is selected here:

https://dxr.mozilla.org/comm-central/source/mail/components/compose/content/MsgComposeCommands.js#6516

  try {
    fontFace = getPref("msgcompose.font_face", true);
    doStatefulCommand('cmd_fontFace', fontFace);
  } catch (e) {}

Function Stack:
loadHTMLMsgPrefs
stateListener.NotifyComposeBodyReady

    if (gMsgCompose.composeHTML)
      loadHTMLMsgPrefs();

InitEditor
gsMsgEditorCreationObserver.observe

However it appears that the mere selection here isn't enough as when the Email
gets focus within a <p> the font selector reverts to "Variable Width"

I raised [Bug 1460808] on the Thunderbird bugzillla for this:
https://bugzilla.mozilla.org/show_bug.cgi?id=1460808

I will need a deeper investigation on which event causes the <font> tags to be
inserted in the unbroken "paragraph" case.
------- Comment #3 From Axel Grude 2018-05-10 23:30:20 ------- 
TECH NOTES 2

When starting to type within the template <div> the font selector defaults back
 to "Variable Width". My current theory is that the event of inserting a <font>
tag is only currently implemented in Editor if the cursor is on the <body>
level. If the cursor is within _any_ other element underneath the body in the
DOM hierarchy the event avoids overwriting the font in case another font was
already specified by the parent element. To be Confirmed.
