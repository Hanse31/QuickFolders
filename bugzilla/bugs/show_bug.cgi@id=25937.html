Hi Axel,

Using QuickPasswords version 3.5

I need backup/restore functionality for QuickPasswords. This will greatly
simplify a couple of tasks I face frequently:-

1 - Rebuilding a local, corrupted installation of Firefox;

3 - Cloning a Firefox installation to a different PC.

I look forward to hearing your comments.

------- Comment #1 From Axel Grude 2015-01-09 06:22:06 ------- 
Accepted the bug.

As I understand it, as backup / restore function would entail in its minimum:

1- INFORMATION STORED

Saving all information of the password manager into a file includes
    Form Submission URL
    host name
    password field name
    user field name 
    user id (encrypted or unencrypted)[*]
    password (encrypted or unencrypted)[*]
    last login date
    last changed date

In a later stage we should allow encrypting the information in this file.

[*] these encrypted field values are actually available but I am not sure
whether they use the master password for encryption so it might or might not be
usable for import into a *fresh* profile

FILE FORMAT

there is a choice of using
1 a) a comma separated files
it would have an option to include a header line so it can be easily opened /
edited in a spreadsheet editor (Excel etc). This is fairly straight forward as
I would choose a manual file target (in the beginning no automatic backup)
1 b) a JSON object file
this is harder to edit externally but has the advantage that various
information can be stored (annotated and web form logins). It is ideal for
storing structured information such as different objects. I might actually
prefer this format but I will play with both.

2 - IMPORT

Two fundamental use cases
2 a) importing into an *empty* profile. Should not be a major problem as there
is no danger to loose information
2 b) importing into an existing profile which already has password information
stored. this is a much more complex use case and should provide a safe way of
resolving duplicates - ideally it should only overwrite passwords when the
change date is higher. This might still not be 100% safe; for safety a backup
file should be generated before overwriting existing information.

Please feedback / reply in-line

Yes, I will see.  editing passwords in bulk isn't necessarily safe so
spreadsheet might actually not be ideal. I tend to think pretty-fied JSON is
best as it is future proof and can easily accommodate additional metadata.

> ​2 - Importing: ​I agree that possible duplicates would be an issue. In my
> own coding, I have usually popped up a warning, offering an replace/clone
> option. This would only be relevant if the record fields were NOT all exact
> duplicates. Otherwise, you are correct in your approach to overwriting existing

alert popups (one per conflict) are typically not ideal for this sort of thing,
so either would have to build a UI (with a list of conflicts) or for the start
only allow modifying (the complete bulk of) passwords which have newer _change_
date. In the first approach this could be an alert

A number of passwords [11] exist but were changed, overwrite?
[Yes] [No]

The other thing that can happen is that the field names change (it would be
likely that the modified date would be different in this case as well). 

So alternatively all modified data could be summed together.

A number of logins were modified [11], overwrite with the newer data?

this would then update field names and passwords provided the restored
information was fresher. 

After this step it might be necessary to display a list of conflicts that could
not be resolved (e.g. different passwords / field names with the same
username/url without a conclusive date difference) - however the likely
assumption here would be that the data in the backup file is outdated.

3.7 prerelease 139 - 1st official version with backup/restore

Here is the very first (!) version with functional backup / restore
functionality. There is quite a big blurb including screen shots on the version
notes site at
http://quickpasswords.mozdev.org/version.html#3.7

Here be a copy for your convenience:

I started to write this after I managed to delete all my sync'ed passwords one
night when I visited a friend and tried to log in from his machine. Somehow
this propagated quickly to my home machine from the Sync store so I had to
restore an older profile. Since Sync doesn't explain where it takes "the latest
password state" from and there is no "yesterday's good logins" I decided it was
time to support a local backup.

However there are some stipulations. First of all I never want QuickPasswords
to be the one simple way to steal your friend's passwords, so the backup /
restore features are protected with the master password. If you haven't set
one, that's too bad, no backup for you. Secondly, we cannot have files full of
unencrypted passwords on some local storage. It's just not safe at all. So by
default, I am using Firefox built in keystore which is a unique personalized
encryption platform generated when you create a new Firefox profile or install
Firefox for the first time. 

This also means that you won't be able to restore the passwords unless you have
a copy of the same keystore. This is stored in a file called key3.db and should
be backed up somewhere to read the passwords again. If you loose or break this
file you will also simply not be able to read your passwords on your Firefox
anymore. these are my conditions for backing up passwords.

All of these stipulations aside, there as a couple of things that the new
backup restore does well (and may be better than others). First of all, when
you restore your passwords, there is one thing it does NOT do: it will not
DELETE all your passwords. Instead it will merge in the passwords, carefully
checking the backed up logins and the time stamps (when was it last changed).
If you want to start from zero, you have to delete all passwords manually.
(Make sure you back up beforehand!)

Modified passwords: Restore will only change a password if it wasn't changed in
the meantime. This way you won't accidentally overwrite a password you may have
changed online into an older version. However, if this is what you actually
want to do with a particular login, [maybe because a password change online
didn't propagate into the web] you can simply delete that one manually before
doing the restore.

File handling: you can store your files wherever you like, but I would still
recommend not putting them on a public folder or shared drive, unless you have
set up correct access permissions. Better safe than sorry!
