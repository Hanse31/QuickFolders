if, e.g., for my work, I have one filter already, when I drag a new message to
my work folder, it will look for the rule I created previously with

Accepted - sounds good. Potentially it is complex, but initially it can be just
based on the target folder. We need to flesh out what it does exactly:

1. if one match is found - it is still important which criteria of the original
email is relevant (that's the whole point of the Template Screen); so I would
not (at least initially) try to do anything clever and parse the existing
filter rules to "guess" what other criteria to use. Also what if the user wants

2. if two or more matches are found - well ideally it would try to consolidate
them all into one, but again, too much work for the one try, so I propose I
display a listbox with just the matching filter names; and the user can select
which one to choose.

3. As regards  higher granularity for the criteria - this is a bit of a sticky
one, as one the one hand I want to keep the template screen "light weight" so
there is not too much thinking & clicking involved; overcrowding with many
checkboxes / options would do just that. On the other hand it would be dead
handy if the user could make her own templates - so maybe that is something for
another bug. As regards "merging" the rules, the existing Templates would

3) CRITERIA DISCUSSION

> "if one match is found - it is still important which criteria of the original

> not (at least initially) try to do anything clever and parse the existing
> filter rules to "guess" what other criteria to use."

In my view, the fight against spam via filters is only effective to the extent
that extra steps can be avoided, at least after one has set some default
behavior.

But I am not envisaging any complex parsing here.

The first idea does not involve any parsing. When you are in Mozilla's normal
"Filter Rules" dialog, the default behavior when you switch from say "Subject"
to "From" is to erase whatever you already started typing in that area.  My
idea here is to override this behavior to retrieve the currently highlighted
email's properties. So if I switch to "From", it will pull the "From" from the
current email. I think this is a far more useful behavior than the default in
TB which simply empties the text, or requiring an intermediate dialog. (I would
also like to see this work with the useful Filtaquilla extesion items, e.g.,
Subject Regex would pull the current Subject, though escaping for regex special
chars. by default.) 

If one wanted a new behavior (e.g., to copy from one highlighted field to
another), I think they can be added as new criteria rules in this Filter Rules
dialog--e.g., "To (based on From)".

Note that your template behavior of setting both "From" and "To" can be
replicated here as well since there is an option (added by Filtaquilla?) for
"From, To, Cc, or Bcc".

But when one really does need to group multiple rules into a template (e.g.,
checking address book and the subject), I think this can be achieved by another
option available on the toolbar folder-specific context-menu, allowing you to
choose which filter pattern is chosen by default (if any) when in non-merge
mode and dragging to that folder.

And when one is not in merge mode, and does not have any filters specified for
non-merge mode, I still think it is useful to be able to be brought to the
filters dialog without needing to go through the Assistant (at least if
changing the criteria in the TB Filter Rules dialog could work as described
above to grab from the current email's contents as relevant by any change or
addition of criteria). Rather than requiring one to temporarily disable the
QuickFilters Assistant, the toolbar folder context menu (and/or a global
preference) could determine whether the Template dialog appears when no filter
is set for merge mode or non-merge mode.

1.6.2, prerelease 21

Here is a very rough version of what I hope will be usable for this feature
request. If a matching filter is found the list of matches is displayed. It is
possible to select _one_ or no entry from the list, then the button [Next] has
to be clicked.

If an item is selected the button text changes to [Edit Filter...]
If no item is selected the button text changes to [Create Filter...]

On [Edit Filter...] the selected filter will be opened for editing and the new
conditions are added automatically. At this stage, I do not check for
duplicated conditions and they shouldn't really happen as a matching filter
should already have moved the mail.

There are 2 known problems at the moment: changes to the Edited Filter are
immediate, so if in Edit Filter the [Cancel] button is clicked the new
conditions are still added and not rolled back. Also, on leaving the dialog,
the edited filter is not highlighted in the filters list. Note that all new
strings are currently hard coded in English.

One thing I would like to start thinking about, how would you provide this
functionality (merge one filter with another) from the filter list dialog? One
problem is that it is already a very overcrowded dialog, so any new tools added
would likely require a toolbar (e.g. something similar to Firefox History /
Bookmarks layout).

Then there are the following problems to solve:

1. how to find out duplicates
2. how to select which duplicates to merge (the new page on the filter
assistant might come into play here or serve as a template)
3. special rules for (re)naming the filter - this is lower priority as I would
display the Edit Filter dialog after a merge operation anyway.

Another thought is that merging filters with same action(s) usually only would
make sense if existing conditions are "or-ed" together. It usually wouldn't
make sense to "extend" a filter that has "and" conditions, e.g.

old rule
"if sender is x and subject is t"
new rule
"if sender is yy"

the UI does not support building
("if sender is x and subject is t") OR ("if sender is yy")
and building a new condition
"if sender is x and subject is t AND if sender is yy"
won't work either.

So for the moment, should merging filters exclude filters with an "and"
condition?
