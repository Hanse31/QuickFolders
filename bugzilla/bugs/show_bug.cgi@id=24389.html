ok, I have scrutinized the code a bit more, and whenever you see the "hc" this
means the version check routine of Mozilla has fucked up.

I put in the "hc" for "hard coded" and its the last resort fallback when the
AddonManager never gives back the version number of my extension. Mozilla have
decided to run version number requests asynchronously and this fuck up is a
direct effect from it. We cannot just check the version number any more we have
to pass in an asyncronous callback function, and god know when that is called -
if ever. I am currently working on a fix that does the "unimportant" checking
whether the extension is  ran for the first time and all the "unimportant"
house keeping (like tidying up old config settings, selecting a proper layout
etc) _after_ thunderbird has decided to pass back the version number. 

At the moment it is broken but I am hoping to have a patched version for you by
tonight.

the latest change) I am running my firstRun test which will check if an update
or a new install have been carried out. During that  I compare the (hopefully
correct) current version number with the one recorded in
extensions.quickfolders.version (that should never be a hc one! - hc stands for
hard coded and it means the version algorithm has f..ed up).

1) If there is _no_ version recorded, then it is a new install, and it should
open the QF homepage.
2) If the version is the same, no action is taken
3) If the version is different, I assume an update - then overwrite the version
number with the current one (e.g. 2.8.3pre12) and display version info and
donation page. (which you can block with these right clicks)

3a) If the version is >=2.7 and previous wasn't then some about:config settings
are tidied up according to extension standards (all configuration settings
should start with extensions.quickfolders); also the new theming engine is
configured for the first time and the "Pimp my tabs"dialog is shown.

So there is a helluva amount of work in that startup routine, and I am quite
cheesed off at Mozilla that they decided to "!bork!" it by removing the simple
way of retrieving the version number. Of course a workaround is to hardcode the
known version number and increase it manually on every build, but I think that
is just bad style.

Anyway, I will upload a fix for these ones as well.

Just a note on this:

behavior by design.

In fact when you re-order the tabs you have to drag them _on top of_ another
tab (not in between) to move them as the Toolbar (background) itself is
"position-agnostic". Only the tabs listen for drop events from other tabs.

So I am not doing any fancy calculation from the perspective of the toolbar in
order to see where the re-positioned tab is being dropped. (if you try to
reposition by dropping exactly in between 2 tabs exactly you will also end up
at the far right, even in "re-positioning mode"). If this ever gets flagged as
a bug I will reject it as WONTFIX.

Likewise, you cannot drop a new folder on top of an existing tab, it is simply
ignored. New folders can only be dragged to the far left or the far right of
the toolbar area (in between two being the same as far right); that's as fancy
as it will get with dragging folders. So drops of FOLDERS on the toolbar itself
are also "postion-agnostic" - to a point.

It still is a huge improvement on the legacy re-positioning system though,
which you can see by selecting "change the order of the bookmarks". A fugly
implementation from the original extension. (I added the animation on the top).
