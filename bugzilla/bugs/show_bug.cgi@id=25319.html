Description:   Opened: 2013-01-25 23:42 
Adding the "Lock" button would help ensure there is no password privacy
violation intentionally or unintentionally in case somebody else is using the
machine for something or more importantly if the machine shared. 

E.g. If a user uses the Password Manager (Master Password protected) for
logging in to website, the manager prompts for master password for the first
time only. Subsequently, if somebody else uses the machine, the person is able
to access the password using the "Copy Password" button, which can then be
pasted anywhere thereby revealing it. And the username is always visible in
password manager anyhow.

Created an attachment (id=7502) [details]
2.6.1 prerelase 53

This version adds a lock icon in front of the [close] button at the bottom
right of the window. It needs to be tested in Linux and Mac yet as I do not
know whether these have different [Close] buttons (or none at all).

The lock acts as a checkbox which remembers its state permanently.
- open: master password will only need to be entered once per session - this is
in sync with the current behavior
- closed lock: resets the authentication session - this happens once you close
the Password Manager; this means the password manager will ask for the master
password next time QP is used. It will also require a password for auto-filling
password information on any login form (as if browser was started without
entering the master password)
- gray closed lock icon: no master password is set yet.

with this implementation it is important to note that there is currently no
"special" permission for the password manager as opposed to automatically
filling forms (which Fx does even when QP is not installed). "Protecting"
Firefox means protecting both authentication forms (autofill) and the Password
Manager window. 

I think a more granular way of configuring security can be achieved by using
the Addon Master Password+, but I must say I found it fairly confusing to use;
also it is non trivial to hide any security affecting settings in an Options
dialog for an Addon that tries to introduce its own security scheme; I would
currently define having more granular security controls as "out of scope" for
the QuickPasswords Addon. 

Also note: There are addons that will allow circumventing or weakening security
within the chrome context, and the AMO policies do not allow much wiggling room
for locking it down much further - e.g. forbidding installation of more addons
or removing addons is hard to lock down for an Addon and being hosted on AMO at
the same time.
