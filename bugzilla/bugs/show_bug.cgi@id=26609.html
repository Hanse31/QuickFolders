Description:   Opened: 2018-11-28 22:32 
A useful utility which has not been updated for many years offered the ability
to export and re-import filters via email:

https://addons.thunderbird.net/en-US/thunderbird/addon/thunderbird-message-filter-imp/

This means that filters can be 'carried around' between installations - rather
than having to transfer files. I used this old add-on to provide me with the
opportunity to 'share' my filter definitions across multiple PC's. At present I
can only share filters by copying the exported filter file, so creating an
email with that file attached ought to be straightforward.

This could be further enhanced by auto-creation export when a new filter
definition is entered, meaning that I can keep my latest filter definitions
automatically updated - if the exported email with filters arrives perhaps it
could automatically bring in the updated filter definitions?

Probably yes. The hard part is importing in a meaningful way.

There is no unique way to identify a filter: the name can be changed so it's
hard to pinpoint whether an imported filter should replace an existing one. The
filters have no unique numbers and can easily be reordered in the filter list.
filters have no ability to store a datestamp (when they were created or
modified)

So even though I have already solved the problem of the import, I am not
actually "merging in" the file - I am just importing everything at the moment
leaving the users to tidy up the mess (My assumption is that they delete their
current filters before import).

I had much more sophisticated of dealing with this in my ADd-on QuickPasswords,
but that was much easier as each login entry stores a date stamp of last
modification and even last use. So there I could safely assume that the "newer"
login could overwrite an existing one. The other advantage was that I could
determine "which one" as each login has a unique username / site URL
combination.

The only thing that is unique about each filter is the combination of search
condition plus the combination of filter actions. This is obviously useless for
replacement as if it matches there is no need to replace the filter anyway. We
are only interested in filters that are changed or added (and maybe use the
name to match) - assuming we can solve the problem of "which one", how do we
determine whether the change has happened on the source machine or on the
target machine - without having a date of the modification?
------- Comment #2 From Andrew 2018-11-29 06:01:54 ------- 
Just a comment to add to the discussion.

As I understand it the filters are saved and restored as plain text, with no
discriminatory ID to identify each filter.

Presumably Quickfilters could enhance its saved filters to add a unique ID for
each filter - stripped off when re-importing. If that ID were a GUID (e.g.)
then its uniqueness could identify previously defined filter definitions
regardless of name changing etc.

Holding that unique key might be an issue, but maybe use of a custom email
header within the filter definition could handle that? Each line of the filter
relates to a specific field within the email, so why not add a new field
definition to hold the filter ID? - not used for comparing values in the email.
The GUID could be assigned when the filter is first created - it wouldn't need
to be shown to the user and if it were in the text stream for the filter it
could be used for identification without impacting standard filter operation.
Users who aren't using  Quickfilters (PRO) would work without being affected -
and these unique ID's would be wholly managed by Quickfilters in its use and
presentation.
