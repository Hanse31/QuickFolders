tabs showing: new mail, unread, and no unread

Running QuickFolders 3.10 on Thunderbird 17.0.2 on Ubuntu 12.04.

I have the "Highlight folders with new mail" option enabled, but the appearance
difference between folders with and without new mail is hardly noticeable. It
appears as if folders with new mail have a sort of gray outline around the tab
label. See attached screen shot.

I thought it was simply a display artifact until I started playing with the
"Highlight folders with new mail" option and did an A-B comparison.

Is there any reason why you didn't simply echo the new mail highlighting scheme
used by Thunderbird itself? They use bold for unread and bold red for new mail.
See second attached screen shot.

Is there a way to customize the highlight appearance?

------- Comment #3 From Axel Grude 2013-01-22 00:24:44 ------- 
Here is the current css rule-set from quickfolders-layout.css; you can add
additional rules in your userChrome.css (profile/chrome folder)

/*** inset shadow on the label ***/
#QuickFolders-Toolbar toolbarbutton[biffState-NewMail].has-unread > label
{
        border-radius: 3px;
        color: #fff;
        box-shadow: inset 0 0 6px #080808;
        padding: 0 3px;
        border-style: dotted;
        border-color: #aaa; 
        border-width: 1px;
}

/*** The following rule didn't work (I would have liked a "green circle" on the
label  ***/
#QuickFolders-Toolbar toolbarbutton[biffState-NewMail].has-unread > label, 
#QuickFolders-Toolbar menu[biffState-NewMail].hasUnread,
#QuickFolders-Toolbar menuitem[biffState-NewMail].hasUnread
{
        list-style-image: url("chrome://quickfolders/skin/ico/newUnread.png")
!important;
}

/*** styles the subfolder popups ***/
#QuickFolders-Toolbar menu[biffState-NewMail].hasUnread ,
#QuickFolders-Toolbar menuitem[biffState-NewMail].hasUnread 
{
        border: 1px dotted #aaa !important;
        box-shadow: 1px 1px 2px 1px #111;
        border-radius: 2px;
}

My thoughts on italics...

Axel Grude wrote:
> italics might be an option as well?

The problem with italics is that it is subtle. More so than bold, and much more
so than bold + red.

In my opinion, styling of the following attributes should go from most visible
to least visible:

new mail
unread mail
normal, and everything else.

When I look at the TB folder tree, I can easily spot folders with new mail from
10 feet away from my monitor. If they were italics instead of red bold text, I
couldn't.


Tony Mechelynck wrote:
> ...italic is already used if the folder isn't empty (or if it is
> empty except for a subfolder).

Axel Grude wrote:
> I never liked italic for "not empty"...what's the point of bookmarking
> an empty folder?

I don't see much value in that particular indicator, though it does appear to
be replicating a styling cue used by TB in its folder tree.

Identifying an empty folder is a low importance attribute, and thus italic is a
good fit.

So this shows new mail in "GMail." The tab does stand out, but not for the
italic text, but because the whole background is green, while all other tabs
just have color stripes.

The colorful, busy nature of the tab style you are using pretty much makes it
an uphill battle to make any indicator stand out.

I get the appeal of using color to coordinate things, but seeing it in use does
make me wonder if there is a better way to accomplish the same thing. Visually,
categorization should be at a more subtle level than status indicator.

For example, use physical positioning. Insert separators between groups of
tabs, and/or put them on different rows in a user controlled way.

If the user needs extensive categorization, then QuikFolders, at least as
implemented as tabs, is going to hit a limit. One can imagine an alternate
implementation for QuickFolders where it is shown as a custom hierarchy of
folders that replace the main folder tree. This would essentially be the same
idea as "favorite folders" that was implemented in earlier versions of TB (no
sign of it in 17), which I tried for a while, but something about it fell
short.

Well that's one case. I meant to reply earlier - it is actually very very
difficult when you are also throw in full themes into the mix. I would like my
interfaces to be usable at least in my favorite themes Nautipolis, Walnut2 and
TT deepdark. So what might look great on Windows with Aero enabled might not
actually be a great choice for Mac OS or Ubuntu with a dark skin. Themes have
no standard mechanism for coloring backgrounds some use background image on a
lower level or various gradients overlapping each other, everybody uses
slightly different selectors so it is extremely difficult to find out the exact
style that needs to be undone. 

Lazy UI designers will resort to attaching an !important rule to any and all of
their styles, I try to avoid this as much as possible so that themes can
override these settings and are able to blend in my own designs better with
their vision / color choice. 

That's also why I chose not to use the "color channel" for this important new
mail status and was looking for something that would override and work with all
custom colors; shadows and sometimes borders are really a good choice here as
they integrate very well with all color choices. A dark shadow (for bright
text) might be invisible on a dark background but it may just be the saving
grace when using a bright one. 

Another problem is that you often cannot even trust the OS-specific colose
(such as menu highlight) as they can be abused by the OS skin for other
purposes. From my own experience I would rather have the user decide for a
color chocie for certain things (like active folder) and pick the color
themselves. A lot of "clever" system color choices make assumptions based on OS
such as bright blue transparent background in Windows Aero - and so they might
think that dark text will always work. They do not take into acocunt that the
Aero engine can be completely restyled (I am using Windowblinds) to look like
Mac or Unix or any fantasy UI (e.g. Startrek). 


> (Although if TB provides adequate access to the DOM nodes or computed CSS, QF
> could still replicate the styles actually in use.)

Computed CSS may help with certain things like font color; but sadly background
appearance is much more complicated than this. You have multiple layers which
can be transparent, so although your label element might be rgba(20,90,180,0.3)
it will still let the background color of a button (which might have a
transparent linear gradient) shine through. that button might be partly
translucent and depend on the hbox (or canvas) that it is one, which might be
bright yellow. So who is to say what resulting "background color" this label
will have? This is a highly complex beast, and the longer I worked with it the
more certain I can say that nothing is predictable.
