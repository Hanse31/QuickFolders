Description:   Opened: 2012-11-01 23:22 
Hi, Axel,
I use a subnote (small screen), which I attach to big screens @home and @work -
so there are 3 resolutions, and therefore 3 different views (plus a huge number
more with changed window widths) of tb and qf.
That means that the tabs are changing the position, so it is not possible to
learn where they are, and I always drag a mail to the right side while the tab
is left at this resolution... Furthermore tabs belonging to each other may be
separated at a certain resolution, due to line wrapping. And every tab placed
not at the end will also change positions of many other tabs... And if number
of eails and the other information, like unread emails, etc is displayed may
also change the position of tabs...
I understand that this is the solution to save most space, but always changing
position...

Thus, grouping of tabs - and the fixed position of these groups, at least
within certain limits - would be great.
Something like fences for the desktop. http://www.stardock.com/products/fences/ 
Prerequisite would be a minimal width - and some good ideas how to use the
space if a big monitor is used. Perhaps also like fences where you can define
different arrangements for every resolution...

An alternative could be categories, but at the moment the use is inconvenient: 
The change of categories requires too much action with the mouse: going all the
way to the most left and upper corner, clicking, then choosing ONE category,
without any visual help (colours), so you have to read before you can select,
then hopefully correctly clicking on the desired category, or starting again...
Most awkward, if you drag a mail and you notice that the desired tabs are not
available in this category. 
Categories should be colour coded, as an option available with one mouseclick
(be a tab), and changeable on-the-fly. And even then, when exceeding a certain
number of tabs, these lose their position...

Greets, Yango

point about the spatial memory but please consider that quickfolders is after
all just a toolbar...

> Thus, grouping of tabs - and the fixed position of these groups, at least
> within certain limits - would be great.
> Something like fences for the desktop. http://www.stardock.com/products/fences/ 

I know and use fences for a while but it is very different to have a whole
desktop area to "play" with and only a small toolbar strip. One feature request
I may point you to is [bug 23039], which proposes inserting line breaks so that
groups of tabs may be in individual lines. I have actually tried to do this for
a while but gave up after spending a couple of hours - I might return to this
bug as well later.

> Prerequisite would be a minimal width - and some good ideas how to use the
> space if a big monitor is used. Perhaps also like fences where you can define
> different arrangements for every resolution...

hmm, that might be overkill, certainly for the majority of users... also like I
said, we have a lot less screen real estate than fences, so I cannot really see
the benefit.

> An alternative could be categories, but at the moment the use is inconvenient: 
> The change of categories requires too much action with the mouse: going all the
> way to the most left and upper corner, clicking, then choosing ONE category,
> without any visual help (colours), so you have to read before you can select,
> then hopefully correctly clicking on the desired category, or starting again...

There is another powerful way you can use categories: open several mail tabs
(thunderbird tabs) and choose a folder from a different QuickFolders category
on each. E.g. you might have a "Work" Tab and a "Friends" Tab and a "Projects"
Tab open. Each of the tabs will be automatically associated with the category
of the chosen folder; you can now simply drag the mail to the tab (if the
category is not displayed at the moment) and it will automatically show the
tabs of the correct category. It is actually more difficult to explain than to
use, please give it a try. One caveat is that this feature was designed when
you could only have a tab in one category, so I need to update this so the Tab
really "remembers" the group in case the current folder there is in a different
one... probably time for creating yet another bug...

PS: I am leaving this bug as unassigned as we apparently haven't got any fixed
requirements.
