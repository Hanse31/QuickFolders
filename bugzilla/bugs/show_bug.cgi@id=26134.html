Description:   Opened: 2016-01-11 15:24 
Suggest allowing the user to specify the tab width. All tabs will always remain
at that width.

Provide global and tab specific options, to enable individual tabs to have a
different width from the global tab width. (Similar to how a tab's ability to
display the email count is managed).

As a bonus, when the user has checked the "Constant Tab Width" option, have the
global tab width value default to the width of the widest currently
instantiated tab. So, for example:
- If the widest tab has an icon and the user has configured Quicktabs to
display email counts in the tabs, then the default global tab width will be
equal to something like this: the name of the tab + a placeholder for
displaying the email counts + the space for the tab icon + any additional
padding that QuickTabs may apply to the tabs.
- If the widest tab does not have an icon but the user has configured QuickTabs
to display the keyboard shortcut on the folder label and to not display email
counts, then the global tab width value will default to the name of the widest
tab + the keyboard shortcut + any padding used in the tab.
- If the user has not yet created any tabs in the QuickTab bar then, if it is
needed at all, the global tab width value can default to something reasonable
like: the width of a tab that will display "Inbox" + the keyboard shortcut + an
inbox icon + the two QuickFolders email counts + any padding.

The current tab behavior in QuickTabs is that the tab widths vary based on the
name of the tab and the font size selected (Shorter names and smaller font
sizes result in narrower tabs, longer names and larger fonts result in wider
tabs). And tabs that display email counts also resize dynamically whenever the
email count changes.

This enhancement will change this tab behavior when the user selects the
"Constant Tab Width" option. This could be implemented by, for example,
allowing the user to specify how many characters wide the tabs should be.

Or, perhaps instead of calling this option "Constant Tab Width" and having the
user calculate the width they want using character count or some other metric,
provide a "Make all of my tabs as wide as the widest tab" check box. Then the
user can just adjust the font size, the mail count display options, and any
other related tab properties, until they obtain the desired tab width.

Those who group tabs may prefer to specify constant tabs widths by tab group
rather than a single global tab width property.

This enhancement would improve usability. QuickTabs allows users to drag and
drop emails. And to support this it already provides the ability to color a tab
and increase font size, to make it easier to find the tab and to hit it when
dragging and dropping. Constant width tabs will enhance this usability by
providing a visual consistency - the tab will consistently look the same (not
longer or shorter based on email count), and the tab will always be in the same
location on the QuickTabs bar (instead of shifting left or right depending on
its email count and the email counts of the tabs to its left).

Use Case: This enhanced usability is particularly noticeable in a use case like
this: 1. The user accesses two separate email accounts via the email client. 2.
The user uses the same set of, say, 8 tabs for each of the two (separate) email
accounts. 3. The 8 tabs for account A are in the top row of the QuickTabs bar,
the 8 identical tabs for account B are in the bottom row of the QuickTabs bar.
With the current QuickFolders tab behavior, in this scenario not only are the
widths and locations of all the tabs constantly changing - shifting left and
right, wider and narrower, but the two rows of identical tabs make the shifting
nature of the user interface more apparent due to the identical tabs on each
rows being staggered rather than aligned in columns.

The least complicated implementation, from the user's point of view anyway,
would be to implement the feature as a "Make all of my tabs as wide as the
widest tab" check box. They just check that and don't need to do any
calculations. And it also avoids the inevitable trial and error user
experimentation that would be required to obtain the desired width when using a
min-tab-width / max-tab-width approach. The user would still have control over
how wide the tabs are, even though they aren't entering a "max tab width"
value. The user could change the tab widths most notably by: 1. Changing the
font size, and 2. Changing the text entered for the tab name (i.e. the number
of characters) in the largest tab. For many use cases this will likely be the
only functionality used, and the modification of individual tab widths will not
be needed.

With this approach, I believe that all that would be needed to enable fine
grained control of individual tab widths is: 1. A check box in the tab's
properties that excludes that individual tab from the global tab width group,
and 2. Adding the capability, in tab properties, to select the font size used
by individual tabs.

User-specified min and max width values would not be not required. Instead, the
user would check the exclude box and then proceed to size the excluded tab
using the methods that are already available in QuickFolders: 1. The amount of
characters they enter for the tab name. 2. The font size they specify in the
tab's properties. 3. Whether to display an icon. 4. Whether to display the mail
counts (unread and total). 5. Whether to display the hot key in the tab.

This approach also avoids the issues discussed in comments #2 & #3. 
(No need to use ellipsis or to move the mail count to the left side of the
tab). With respect to #2, the folder name will never be too long to fit in the
tab because the tab width is still being calculated by QuickFolders based on
the content of the tab, and not by user-entered max/min values. 

#3 is addressed by making a few changes to the way QuickFolders displays the
mail counts in a tab: 1. If a folder is configured to display the mail counts
then, in order to prevent the folder width from changing dynamically each time
the mail count changes, the folder shall always have a value displayed for mail
counts. When the mail count is zero it should display "0" or a blank. 2. To
manage the display of arbitrarily large mail counts, when the number of emails
in the mail count exceeds the number of digits available to display the count,
then the tab can display "99+", or "999+", or "9999+", etc (or "++", "+++",
"++++", etc, which will take up one less character of valuable tab-width real
estate). 3. Allow the user to specify how large a mail count to display in the
tab before limiting it and using the "+". 

The question of centering and right/left justifying the tab name will also need
to be decided. For a tab with a short name that has now grown much wider in
order to be the same width as the widest tab, shall QuickFolders determine
where to place the name within the tab, or will it allow the user to specify
where the name shall be displayed in the tab - centered, right justified, or
left justified? Considering some of the basic formatting capabilities of
spreadsheets will be helpful with this question.

Also, what method shall be used to pad the shorter tabs to make them the exact
width of the largest tab? And how will the tab widths be measured? If these two
things can be done using the existing tab attributes (characters, text fields,
icons, etc), then the enhancement will be easier to implement than if a
completely new metric, that uses attributes and data that QuickFolders doesn't
currently use, is required for measuring and padding.

the problem isn't the users calculating, it's _me_ constantly finding out the
widest tab width (and this is also dynamic so is affected by unread email
counts) and then applying to the other tabs. That's the calculations I want to
avoid. They will complicate my code and have a negative effect on performance.
It is usually easier to use CSS rules, so if this can be done with CSS (min /
max width should definitely work) then that's the preferred and most performant
route.

> For many use cases this will likely be the
> only functionality used, and the modification of individual tab widths will not

But only by the small percentage of users who want all tabs to have the same
size. Let's look at how TabMix plus solves the problem.

https://i.imgur.com/ZXXBXPF.png

They have a minimum and maximum width (in pixels) and a checkbox (Tab width
fits to title)

When the checkbox is off Tabs stay dynamic within min / max boundaries
depending on text.
When the checkbox is on (which is pretty much QuickFolders behavior) the
maximum value is ignored and the First Tab width applied to everything.
Considering that Tabmix has a HUGE amount of configurability (too complicated
for my taste) you can probably appreciate why they didn't opt for doing "behind
the scenes width calculations".

Having said that, there might be an option using the new css "calc()" function
(if there is also such a thing as "max()". We can start throwing some proof of
concepts around via editing userChrome.css. Hope you are familiar with it. :)

http://www.sitepoint.com/css3-calc-function/

Reading that I cannot find a CSS function that takes the width of other
elements into account, [something like max(children.width)]

I also played around briefly with 

http://stackoverflow.com/questions/6310632/html-list-element-sharing-the-parent-width-into-equal-parts

but the display as table doesn't work very well for floating buttons (As these
are). Instead, if you add the following rules to your userChrome.css you can
immediately achieve something like what you want:

#QuickFolders-FoldersBox toolbarbutton {
  min-width: 10em;
  max-width: 10em;
}

looks like this on my system: https://i.imgur.com/cjkgETh.png

(note that I have a separator before the folder "Allan", which moves the top 6
tabs to the right, so you might want to not use any separators. 

So in a nutshell I am very likely not going to be able to "calculate the widest
tab" based on contents etc as it is a lot of calculating & hassle for something
that can be so easily / low-tech achieved via min/max.
