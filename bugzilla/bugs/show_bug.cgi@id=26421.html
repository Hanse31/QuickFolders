Description:   Opened: 2017-09-16 17:40 
Using smarttemplate4 

E.g.:

Date: Sat, 16 Sep 2017 07:42:16 +0700

Using the following in reply field:

%X:=sent%On %a%, %b% %d%, %Y% at %I%:%M%:%S%%p% %date_tz%

produces:

On Sat, Sep 16, 2017 at 03:42:16AM +0300

where date/time and "+0300" are *my* timezone, not the one found in header.

You're welcome!

Further investigation shows there is an optional switch with %X:=sent%

%X:=sent(UTC)%

which supposedly forces the timezone of the sent time stamp to UTC. Maybe this
can be extended using something likee

%X:=sent(senderTZ)%

(for forcing sender's timezone) in the current line. This willl necessitate
parsing the timezone string manually to find out the time zone and doing
appropriate calculations locally, which may introduce some errors (retrieving
the offsets from the web is not a good option for performance reasons)

Reason for complexity: the JavaScript date objects supports functions like
getUTCFullYear() and getUTCMonth(), getUTCDate(), getUTCHours() and
getUTCMinutes() out of the box. The other "built in" functions 
getFullYear() and getMonth(), getDate(), getHours() and getMinutes() 

always work from the local time zone. I would have to write my own overloaded
functions for this, or maybe find a way to temporarily overwrite the local time
zone in order to generate remote times (which may be impossible or high risk)
as a quick hack.
