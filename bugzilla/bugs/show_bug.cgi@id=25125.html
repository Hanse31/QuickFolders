Description:   Opened: 2012-09-04 16:39 
Where I work, our email server is now hosted externally by Microsoft
(outlook.com).  Any operation on those external folders requires up to 10
seconds while Thunderbird sends login credentials to the outlook.com server.

On two recent occasions, I selected between three and 12 (or so) emails from an
inbox hosted on the external server, then dragged them to a QF tab (local
storage folder).  In the status bar at the bottom of TB, I could see that TB
was sending login credentials to the external server, but QuickFolders
completed its "move" operation before the login to the external server was
complete.

As a result, the email was removed from the external server (as expected), but
those emails in the local folder were 0 bytes in length; their contents were
gone.

After the second occurrence, I disabled QF.  Since then, I have been moving
multiple emails in TB the "old fashioned" way and so far have not experienced
any missing email content; TB seems to wait for the login credentials to be
exchanged before executing the move.

TB has always been configured to keep the external server folders in-sync
locally.

Does this sound like a plausible QF issue?

If you can still see the headers (and haven't deleted the empty mails), you
should try rebuild the folders - you can either use  the QF context menu mail
commands > repair folder, or right-click the folder in the folder tree, select
"properties..." and click the [repair folder] button. There is a good chance
you will get their contents back like this.

On first look, it doesn't sound like it, I am using the messagecopy service of
Thunderbird for moving / copying emails, here is the relevant code:

var cs = CC["@mozilla.org/messenger/messagecopyservice;1"]
  .getService(CI.nsIMsgCopyService);
targetFolder = targetFolder.QueryInterface(CI.nsIMsgFolder);
cs.CopyMessages(sourceFolder, messageList, targetFolder, !makeCopy, null,
msgWindow, true);

Obviously the folder part seems to work (the mails arrive in the correct
folder). Normally Tb would not delete the original emails while moving until it
the copy operation was complete.

Using this interface, there seem to be no special precautions to take when the
messages are copied from one server to another; AFAIK this is the only
interface that can be used to copy them, so QuickFolders does not do anything
special in comparison to Thunderbird itself. I will look investigate it with
the #maildev IRC group to see if it is possible to check whether the (from)
server is online, and whether Thunderbird does it in it native code when you
use the folder tree to do a move/copy operation.
------- Comment #2 From Dave Sp 2012-09-06 02:45:46 ------- 
Answers to your questions:

Account is IMAP.

Although I did not try to repair the folders, I used vi (via Cygwin) to examine
the mail folder files directly, and at the end of the file there was a "From -
DDD MMM dd HH:MM:SS YYYY", followed by a blank line, for each missing email. 
No other header or body lines.  The emails were also missing from the
originating folder.

The "Local Folders" are just locally-stored copies of the server emails that I
want to keep.  (I presume most folks locally store at least some emails.)

Another note is that the connection to the outlook.com server is not
persistent, even though TB is set to check for new mail every 60 seconds. 
Accessing other inbox folders on the outlook.com server often requires TB to
re-login to the server even if it's been well under 60 seconds since the last
server folder was checked.

So maybe this lack of a persistent connection could be a factor.
