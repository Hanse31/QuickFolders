Description:   Opened: 2014-12-25 03:28 
Is there any way to have the right-click 'retrieve a password' option
automatically copy, and paste if possible, the password from the matching
entry, and perhaps not even open the password manager box?  A short message
advising of which action was taken might be helpful.

Another nice addition would be a right-click 'retrieve a User ID' option.

As long as you are in the password manager this is a single click operation, so
I don't exactly understand what you mean when you say to "retrieve...the
password from the matching entry". In the example on this screenshot you can
see there can be a multitude of logins available; currently I am not recreating
the whole list in a separate UI (but conveniently use the password manager for
this, as it also offers the master password protection).

attach a drop down menu with the matching logons to the toolbar button for next

it will also need to sort these by date because with drop down menus there is

rather have these choices on the toolbar button.

"A few steps?" It does this already, via a one click operation. the first
(green arrow) button on the Password Manager does precisely that. So does
double-clicking the login on the Password Manager.

> 
> Technically, the right click 'Retrieve a Password' does not actually do the
> retrieval, it opens the password manager (or Saved Password Editor, which I
> use) and then allows the user to do the retrieval.

Yes, like I said it currently requires one more click. So you have only 2 (!)
clicks altogether. 
1 - right-click and choose retrieve a password
2 - click the login button. It automatically fills both pwd/username fields for
you.

3 - potential exceptions:
3a) the web page has changed the name of the login field names, in which case
QuickPasswords cannot reliably fill them 
in that case I show a notification and add "insert username" / "insert
password" in the context menu. Once you use these menu items they are removed
(for security reasons) and a sliding notification asks to record the field name
change so QP can fill the password automatically in the future. Some login
pages use randomized form field names in order to completely disable password
management, there is nothing that can be done to circumvent this without adding
security risks. (e.g. I refuse to "guess" where the password field is as this
might have been added by malicious script)

3b) there is no <form> tag on the login page. Some pages do this, I guess again
to prevent people from using password managers. Currently I do not have a
technical solution for this - also neither Firefox nor the highly recommended
"Saved Password Editor" extension are able to help work around this in order to
even store the password. Password management requires a well defined login
form.

>  The fact that it does a screening on the URL is a very handy feature.

Since release 2.8 it does more - it highlights your most recent login so you do
not have to choose from the list (see my previous screenshot). I think this is
also important as I have many identities when I am working on the internet.

> And, if this were possible, then to copy the procedure for the ID field.
> 
Again, I don't quite understand what you exactly want to do - Maybe you can
describe it step-by-step (please also only do *one* feature request per bug)?

Bear in mind that eliminating the password manager dialog is a big piece of
work as I need to extract a lot of logic from the Mozilla code base to do this.
Using the password manager dialog was a great shortcut for not writing 100s
additional lines of code and posing potential security risks.
