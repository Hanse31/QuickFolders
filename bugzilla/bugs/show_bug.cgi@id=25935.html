QuickPasswords 3.5 prerelease 13

Fixes the issue - this was caused by re-scoping a variable following the bug
fixes in [Bug 25909]. 

This version also has improved robustness in the SSO change processing -
previous versions had early exits caused by multiple login names on the same
site which made it necessary to run the SSO change repeatedly. This is now
fixed as far as possible. If QuickPasswords fails to change a password due to
the system (throws an exception) this will be now logged in the error console
and won't cause a "short-circuit" in the loop that implements the logon
changes.

The SSO process should be further improved as it is currently clunky in that it
needs the correct domain filter set (before clicking the "Select all entries
that match the password" button) in order to reach all affected SSO login
entries. This could be further automated for ease of use (note to self or
users: raise a separate bug on this)
